# RK3588 + RK1822 平台 — 模型选型与基准测试报告

**平台：** Rock 5B / OrangePi 5+ 类开发板  
**最后校准：** 2026-06-20。本文件原地更新。

---

## 硬件架构

本设备包含两条独立的 NPU 计算路径：

| 组件 | 芯片 | 接口 | 服务 |
|---|---|---|---|
| **宿主 SoC** | Rockchip RK3588（4×A76 + 4×A55） | 片上 | RKNPU3（3 TOPS）— rkllm3-server port 18001 运行 LLM 推理 |
| **NPU 协处理器** | Rockchip RK1822（PCIe，Device 182a） | PCIe `0003:31:00.0` | uvicorn server port 18080 运行 embedding 推理 |

各模型和基准测试均标注了实际运行的芯片。

## 跳过的测试维度

以下维度对 `qwen3-vl-2b-rk3588` 永久跳过：

| 维度 | 原因 |
|---|---|
| `general_ability` | ctx=768 token 上限阻止 few-shot GSM8K；MMLU/HellaSwag 不可用（HuggingFace 设备不可达，缓存不完整）。参考：GSM8K 非正式跑分 66%（2026-06-19，未校准）。 |
| `prefill_decode` | ctx=768 下样本量少，结果不可靠 |
| `conditioned` | ctx=768 不足以支持多轮系统提示 |
| `scenarios` | ctx=768 不足；需先通过 conditioned |
| `accuracy` | VLM 视觉接地不支持纯文本 task_type |
| `stability`、`concurrency` | 长时间测试在 NPU 上存在孤儿锁风险，不切实际 |
| `embedding`、`rerank`、`asr`、`ocr` | 能力类别不匹配 |

---

## 已知局限

- **上下文上限 768 token** — RKLLM runtime 硬性限制；有效限制了响应长度，并阻止 few-shot benchmark。
- **NPU 孤儿进程锁死** — `rknn3_transfer_proxy` 卡死后会锁死 NPU 硬件；需 reboot 恢复，目前无软件解决方案。
- **HuggingFace 不可达** — 设备无法直接访问 huggingface.co，只有预缓存数据集可用。GSM8K（openai/gsm8k）已缓存；MMLU 和 HellaSwag 未缓存。
- **双芯片架构** — LLM 和 embedding 分别在物理上独立的 NPU 芯片上运行（RK3588 RKNPU3 vs RK1822 PCIe NPU），各自的性能和可用性相互独立。
